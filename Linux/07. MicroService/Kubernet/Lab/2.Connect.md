# Kết nối Worker với Master
**Bối cảnh hiện tại**: Máy Master (172.16.66.84) đang chạy API Server và etcd. Hai máy Worker (172.16.66.82, .83) đã có Kubelet và Containerd chờ lệnh, nhưng Kubelet chưa biết Master ở đâu và cũng chưa có quyền để nói chuyện.

## Lý thuyết
API Server sẽ có các cơ chế bảo mật của mình. Các Worker Node không thể cứ thế kết nối đến Server tự động mà thay vào đó chúng sẽ cần có 1 phương thức xác thực bảo mật nào đó. Ở đây chúng ta có mTLS(chứng chỉ 2 chiều).

Nhưng làm sao máy U82 có chứng chỉ khi nó mới được bật lên? Đó là lúc quy trình TLS Bootstrapping diễn ra qua 4 bước:

![](/Anh/Screenshot_1049.png)
- Discovery (Dò tìm & Chống giả mạo - MITM): Khi bạn chạy lệnh `join`, nó có kèm một mã `ca-cert-hash`. Máy `U82` dùng mã băm (hash) này để xác minh: "Đúng là máy `U84` rồi, không phải một thằng hacker nào đó đang giả mạo Master để lừa lấy dữ liệu của mình".
- Authentication (Xác thực ban đầu): Máy `U82` trình ra cái --token (như một mật khẩu dùng một lần). API Server nhìn thấy token đúng, cho phép `U82` bước qua cửa phòng bảo vệ.
- CSR - Certificate Signing Request (Xin cấp căn cước):
Lúc này Kubelet trên máy `U82` tự động tạo ra một chìa khóa riêng (Private Key) nằm chết trên ổ cứng của nó, đồng thời tạo một tờ đơn xin cấp chứng chỉ (CSR) gửi lên API Server. Tờ đơn ghi: "Hãy ký xác nhận tôi là node k8s-worker-1".
- Approval & Provisioning (Phê duyệt và Cấp phát):
Trong kiến trúc Kubeadm, Controller Manager trên Master tự động duyệt tờ đơn (CSR) này, ký bằng CA của cụm, và gửi Chứng chỉ (Client Certificate) trả lại cho máy `U82`. Kubelet lưu chứng chỉ này vào file `kubelet.conf` và từ nay về sau dùng nó để nhận lệnh.

## Thực hành
Ở bài trước, chúng ta đã có câu lệnh `join`, tuy nhiên để chắc chắn, có thể lấy lại câu lệnh `join` như sau:
```
kubeadm token create --print-join-command
```
```
kubeadm join 172.16.66.84:6443 --token qkv0p6.zkcht36xcdj570m4 --discovery-token-ca-cert-hash sha256:3e409ebe0c7e4fb8ff21bd560fc31d39418c064a35fed3da28e5b6ca7ac4d115
```

### Lúc này các chứng chỉ sẽ nằm ở đâu trên máy Worker?
Kiểm chứng file lưu chứng chỉ
```
root@MANH-U82:~# ls -l /var/lib/kubelet/pki/
total 12
-rw------- 1 root root 1118 Mar 23 17:40 kubelet-client-2026-03-23-17-40-02.pem
lrwxrwxrwx 1 root root   59 Mar 23 17:40 kubelet-client-current.pem -> /var/lib/kubelet/pki/kubelet-client-2026-03-23-17-40-02.pem
-rw-r--r-- 1 root root 2295 Mar 23 17:40 kubelet.crt
-rw------- 1 root root 1679 Mar 23 17:40 kubelet.key
```
Bạn sẽ thấy file `kubelet-client-current.pem`. Đây CHÍNH LÀ tấm thẻ căn cước mTLS mà máy U84 vừa cấp cho máy U82. Kubelet dùng nó để mã hóa kết nối HTTPS với API Server

### Đọc file cấu hình
Kubeadm đã tự động sinh ra 1 file cấu hình để Kubelet biết đường nói chuyện với Master. 
```
cat /etc/kubernetes/kubelet.conf
```

![](/Anh/Screenshot_1051.png)

File YAML này chứa `server: https://172.16.66.84:6443` và trỏ thẳng tới các chứng chỉ. Nếu không có file này, Kubelet sẽ mất trí nhớ và bị văng ra khỏi cụm, gây lỗi.

### Kiểm tra file csr trên Master
Ngay khi máy U83 và U82 báo join thành công, hãy quay về máy U84 để thực hiện kiểm tra. Cần nhanh tay vì quá trình này diễn ra tự động và rất nhanh. Sau khi ký xong, K8S sẽ tự động dọn dẹp các chứng chỉ này
```
kubectl get csr
```
```
root@k8s-master:~# kubectl get csr
NAME                                                   AGE   SIGNERNAME                                    REQUESTOR                 CONDITION
node-csr-Q1w2e3R4t5Y6u7I8o9P0a1S2d3F4g5H6j7K8l9Z0x     12s   kubernetes.io/kube-apiserver-client-kubelet   system:bootstrap:qkv0p6   Pending
node-csr-A1b2C3d4E5f6G7h8I9j0K1l2M3n4O5p6Q7r8S9t0u     2m    kubernetes.io/kube-apiserver-client-kubelet   system:bootstrap:qkv0p6   Approved,Issued
```
- **NAME**: Trong K8s, tên tờ đơn CSR của Node luôn bắt đầu bằng `node-csr-` kèm theo một chuỗi mã băm ngẫu nhiên.

- **AGE**: Thời gian kể từ khi máy Worker nộp đơn xin cấp chứng chỉ lên API Server.
- **SIGNERNAME**: kubernetes.io/kube-apiserver-client-kubelet. Đây là thông tin nói cho hệ thống biết rằng: "Tờ đơn này xin cấp chứng chỉ để làm Kubelet Client, và nó cần được ký bởi CA của API Server".
- **REQUESTOR**: Hãy chú ý cái tên `system:bootstrap:qkv0p6`. Lúc này máy Worker chưa có tên chính thức trong hệ thống, nên nó dùng chung một tài khoản tạm thời dành riêng cho việc bootstrap. `qkv0p6` chính là 6 ký tự đầu tiên trong cái --token mà bạn đã dùng ở lệnh `kubeadm join (qkv0p6.zkcht36xcdj5...)`. API Server nhìn vào đây để xác thực thân phận ban đầu của Worker.

- **CONDITION**:
  - **Pending**: Tờ đơn vừa nộp lên, Controller Manager đang kiểm tra xem token có hợp lệ không.
  - **Approved,Issued**: Controller Manager gật đầu, ký chứng chỉ và đẩy ngược lại cho máy Worker. Lúc này Worker chính thức trở thành người nhà và lưu chứng chỉ vào file `/var/lib/kubelet/pki/kubelet-client-current.pem`

### Kiểm tra các node trên Master
Hãy cùng xem các node đã thực sự join được tới Master chưa
```
kubectl get nodes
```
```
root@k8s-master:~# kubectl get nodes
NAME           STATUS     ROLES           AGE   VERSION
k8s-master     NotReady   control-plane   22h   v1.29.15
k8s-worker-1   NotReady   <none>          20h   v1.29.15
k8s-worker-2   NotReady   <none>          20h   v1.29.15
```
Cả 3 máy (k8s-master, k8s-worker-1, k8s-worker-2) đã nhận diện nhau, nhưng TẤT CẢ đều đang ở trạng thái NotReady.

Tại sao lại như vậy? Hãy hỏi API Server xem lý do thực sự là gì. Chúng ta mô tả chi tiết máy U82:
```
kubectl describe node k8s-worker-1
```
