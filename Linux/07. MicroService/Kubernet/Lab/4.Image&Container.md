# Image và Container trong K8S
Trước khi dùng `kubectl` búng tay tạo ra hàng loạt ứng dụng, chúng ta phải đi xuống tận đáy của hệ điều hành để xem một ứng dụng thực sự được K8s "nhào nặn" ra sao.

## Image 
### Khái niệm 
Imgae chính là template để tạo ra các container trong pod

Image không phải là một file thực thi đơn lẻ hay một máy ảo. Theo chuẩn OCI, một Image thực chất là một tập hợp các thành phần sau:
- Các lớp file system: Image được xây dựng dựa trên công nghệ UnionFS. Thay vì gộp chung, mỗi khi bạn cài thêm một phần mềm vào Image, nó tạo ra một lớp mới đè lên lớp cũ. Các lớp này có tính chất Read-only và Immutable
- Image Manifest: Đây là file siêu dữ liệu. Nó định nghĩa danh sách các layers tạo nên cấu trúc thư mục, kiến trúc CPU hỗ trợ (amd64, arm64), và các tham số khởi động mặc định (ENTRYPOINT, CMD, ENV)

### Nguồn gốc của Image
Kubernetes và Containerd không tự sinh ra Image. Khi có lệnh chạy ứng dụng, chúng phải tải (pull) Image từ các hệ thống lưu trữ tập trung gọi là Container Registries.

Định dạng một đường dẫn Image (Image URI):
Một cái tên ngắn gọn như `nginx:latest` thực chất là một dạng viết tắt. Cấu trúc đầy đủ của nó là:
`[REGISTRY_DOMAIN]/[REPOSITORY_NAME]/[IMAGE_NAME]:[TAG]`
(Ví dụ: `docker.io/library/nginx:latest`)

Các loại Registry phổ biến:
- Public Registries (Công cộng): Docker Hub (docker.io), GitHub Container Registry (ghcr.io), hoặc kho gốc của K8s (registry.k8s.io - nơi chứa các image lõi như kube-proxy, kube-apiserver).
- Private Registries (Nội bộ): Trong môi trường doanh nghiệp, bạn sẽ dựng các Registry riêng (như Harbor, GitLab Registry, AWS ECR) để bảo mật mã nguồn. Containerd trên máy Worker bắt buộc phải được cấu hình xác thực (Tài khoản/Mật khẩu) mới có thể pull được image từ đây.
## Container
Khi Kubelet ra lệnh cho Containerd "chạy" một Image, Containerd sẽ lấy các lớp Read-only của Image đó, gắn thêm một lớp Read-Write (Đọc-Ghi) mỏng dính lên trên cùng. Tập hợp này trở thành hệ thống file (rootfs) cho Container.

Tuy nhiên, Container không phải là máy ảo (VM). Nó không có nhân hệ điều hành riêng. Container bản chất 100% là một tiến trình (process) bình thường của Linux Kernel, nhưng bị cô lập bởi 2 công nghệ cốt lõi:
- Namespaces: Giới hạn tầm nhìn của tiến trình.
  - PID Namespace: Tiến trình trong container tưởng nó là PID 1 (tiến trình gốc), nhưng trên máy chủ thật nó có thể là PID 25430.
  - Network Namespace: Nó có card mạng ảo riêng, bảng định tuyến riêng.
  - Mount Namespace: Nó chỉ nhìn thấy hệ thống file được cấp từ Image, không thấy ổ `C:` hay `/` của máy chủ vật lý.
- Cgroups: Kubelet báo cho Containerd cấu hình Cgroups để ép tiến trình này lên Linux Kernel. Ví dụ: Cgroups sẽ khóa tiến trình này lại, chỉ cho phép dùng tối đa 256MB RAM và 0.5 core CPU. Nếu tiến trình dùng vượt mức, Kernel sẽ bắn tín hiệu SIGKILL (OOMKilled) giết chết tiến trình ngay lập tức.

## Image vs Container
- Image: Nó là một tập hợp các file nén (tarball) chứa mã nguồn ứng dụng, thư viện (libs) và một hệ điều hành thu nhỏ. Nó read-only. Nó nằm im trên ổ cứng.
- Container: Khi hệ điều hành lấy cái "Bản vẽ" đó, bung nó ra RAM, cấp cho nó CPU, và dùng công nghệ Namespaces/Cgroups của Linux để nhốt nó lại -> Lúc đó ta có một Container. Container là một tiến trình đang chạy.

Nếu bạn gõ lệnh bảo K8s tạo một ứng dụng Nginx, K8s không hề tự tay tải file hay chạy ứng dụng. Chuỗi quyền lực diễn ra như sau:

- Bạn gõ `kubectl` -> Gửi lệnh cho API Server
- API Server ghi vào etcd -> Scheduler tìm thấy máy U82 đang rảnh, bèn điền tên máy U82 vào.
- Kubelet nhận được tin báo: "Ê, có đơn hàng chạy Nginx".
- Kubelet không biết làm thế nào để chạy Nginx. Kubelet xoay sang gọi Containerd và ra lệnh: "Tao cần 1 container Nginx, mày lo từ A-Z cho tao".
- Containerd mới đi tải Image Nginx về mạng, bung nén, và móc nối với Linux Kernel để chạy thành Container

## Công cụ crictl: 
Ngày xưa dùng Docker, bạn gõ `docker ps` để xem container. Giờ K8s dùng Containerd, lệnh docker không còn tác dụng. Thay vào đó, K8s cung cấp công cụ crictl (CRI CLI) – một công cụ giao tiếp thẳng với Containerd.

## Thao tác với crictl
Thay vì sử dụng `kubelet`, chúng ta sẽ sử dụng `crictl` để xem cách mà K8S thực sự vận hành
### Thiết lập crictl trên Worker
Chỉ định cho crictl biết cái socket của Containerd nằm ở đâu:
```
# Định cấu hình endpoint cho crictl kết nối với Containerd
crictl config runtime-endpoint unix:///var/run/containerd/containerd.sock
crictl config image-endpoint unix:///var/run/containerd/containerd.sock
```

### Tự tay cài image với crictl
Ở đây chúng ta sẽ dùng nginx cho cơ bản
```
crictl pull nginx:alpine
```
Bạn sẽ thấy nó tải về từng dòng, đó chính là nó đang kéo từng Layer theo chuẩn UnionFS

Kiểm tra
```
root@k8s-worker-2:~# crictl images
IMAGE                                   TAG                 IMAGE ID            SIZE
docker.io/library/nginx                 alpine              d0c7807749103       26MB
ghcr.io/flannel-io/flannel-cni-plugin   v1.9.0-flannel1     7bf3bb53c74b2       5.14MB
ghcr.io/flannel-io/flannel              v0.28.1             5d0f6aa70335d       34.1MB
registry.k8s.io/kube-proxy              v1.29.15            f71614796eb76       28.8MB
registry.k8s.io/pause                   3.8                 4873874c08efc       311kB
```
### Khám phá file Manifest 
Thử nghiệm kéo một Image thực tế từ Public Registry (Docker Hub) về ổ cứng của máy U82:
```
crictl pull docker.io/library/redis:alpine
```

Khám phá file Manifest (Metadata) JSON của Image vừa tải để xem K8s lấy thông số khởi động từ đâu:
```
crictl inspecti docker.io/library/redis:alpine
```

Bạn không hề tạo ứng dụng nào, nhưng lệnh `crictl ps` lại trả về một loạt các Container đang chạy! Hãy nhìn vào cột IMAGE hoặc NAME. Bạn sẽ thấy:
- Container của `kube-flannel`: Kubelet đã âm thầm yêu cầu Containerd chạy nó

- Container của `kube-proxy`: Kubelet chạy nó để chuẩn bị làm nhiệm vụ định tuyến

### Vậy nếu khi 1 container bị lỗi thì sao?
Nếu khi có 1 container bị lỗi thì liệu có gây ra lỗi toàn bộ hệ thống không? 

Điều này không thể xảy ra do khi kubelet theo dõi và phát hiện có container lỗi, nó sẽ truyền lệnh tới containerd, ép nó tạo ra 1 container mới. Bây giờ chúng ta sẽ cùng thử:

Trước tiên chúng ta thực hiện kiểm tra 
```
crictl ps
CONTAINER           IMAGE               CREATED             STATE               NAME                ATTEMPT             POD ID              POD
e6cd3087124cb       5d0f6aa70335d       46 hours ago        Running             kube-flannel        0                   dfd36859e3402       kube-flannel-ds-45gm5
c188c797d922f       f71614796eb76       2 days ago          Running             kube-proxy          0                   77bc9d7423f18       kube-proxy-ggbr2
```
Sau đó, ví dụ chúng ta xóa kube-proxy
```
root@k8s-worker-2:~# crictl rm -f c188c797d922f
c188c797d922f
c188c797d922f
```
Kiểm tra lại
```
root@k8s-worker-2:~# crictl ps
CONTAINER           IMAGE               CREATED             STATE               NAME                ATTEMPT             POD ID              POD
deba76809a559       f71614796eb76       3 seconds ago       Running             kube-proxy          1                   77bc9d7423f18       kube-proxy-ggbr2
e6cd3087124cb       5d0f6aa70335d       47 hours ago        Running             kube-flannel        0                   dfd36859e3402       kube-flannel-ds-45gm5
```
Như các bạn có thể thấy, mặc dù đã xóa nhưng khi kiểm tra lại, chúng ta lại thấy kube-proxy tự động được tạo mới
