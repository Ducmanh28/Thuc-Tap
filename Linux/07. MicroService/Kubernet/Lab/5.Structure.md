# Core Structure, Component and OS Process
## Core Structure
Kubernetes không phải là một nguyên khối. Nó là một hệ thống phân tán (distributed system) bao gồm các daemons và binaries tương tác với nhau qua mạng.

## Control Plane Components (Vận hành trên Master Node - 172.16.66.84)
Các thành phần này chịu trách nhiệm duy trì Global State và ra quyết định điều phối. Trong kiến trúc deploy bằng kubeadm, chúng chạy dưới dạng Static Pods.

- ***kube-apiserver***: Frontend duy nhất của K8s Control Plane. Cung cấp REST API xử lý các tác vụ CRUD. Mọi component khác (bao gồm cả kubelet trên worker node) đều phải giao tiếp qua TLS termination của apiserver. Nó xử lý Authentication, Authorization (RBAC), Admission Control và là component duy nhất có quyền connect trực tiếp tới etcd.

- ***etcd***: Hệ quản trị cơ sở dữ liệu Distributed Key-Value store, sử dụng thuật toán đồng thuận Raft. Lưu trữ toàn bộ Cluster State và Configuration. Dữ liệu được serialize dưới định dạng protobuf hoặc JSON.

- ***kube-scheduler***: Lắng nghe event từ apiserver về các Pod mới chưa được gán thuộc tính nodeName. Thực thi 2 phase: Filtering (áp dụng các Predicates để loại bỏ node không đủ tài nguyên) và Scoring (áp dụng Priorities để chấm điểm node phù hợp nhất). Cuối cùng, bind Pod vào Node được chọn.

- ***kube-controller-manager***: Chứa các background threads chạy vòng lặp điều khiển (Control Loops / Reconciliation loops). Nó liên tục watch (theo dõi) trạng thái hiện tại (Actual State) từ apiserver và thực thi các API calls để đồng bộ hóa với trạng thái mong muốn (Desired State). (VD: ReplicaSet Controller, Node Controller).
## Worker Node Components (Vận hành trên Worker Node - 172.16.66.82, .83)
Các component tại đây chịu trách nhiệm cấp phát tài nguyên Compute, Memory và Network cho Workload.

- ***kubelet***: Agent chính chạy trên mỗi node. Khác với control plane, kubelet chạy trực tiếp dưới dạng một systemd daemon của Linux OS. Nó đăng ký Node với apiserver, nhận PodSpecs và tương tác với Container Runtime qua giao thức gRPC để đảm bảo các containers đang ở trạng thái Running và Healthy.

- ***Container Runtime*** (Containerd): Phần mềm thực thi CRI (Container Runtime Interface). Quản lý image pulling và gọi down cấp thấp hơn để setup Linux Namespaces và Cgroups cho container process

- ***kube-proxy***: Network routing daemon chạy dưới dạng DaemonSet trên mọi node. Nó watch các object Service và EndpointSlice từ apiserver để duy trì network rules trên host OS. Đảm bảo L4 load balancing cho traffic TCP/UDP hướng tới các Pod backend

## Deep-Dive
### Inspect Control Plane Configurations
Các component của Control Plane chạy dưới dạng Static Pods, nhưng cấu hình thực sự của chúng nằm ở các file YAML mà `kubelet` đọc trực tiếp từ disk

- Cấu hình khởi động của Api Server
```
root@k8s-master:~/.openclaw# cat /etc/kubernetes/manifests/kube-apiserver.yaml | grep -E "authorization-mode|etcd-servers|service-cluster-ip-range"
    - --authorization-mode=Node,RBAC
    - --etcd-servers=https://127.0.0.1:2379
    - --service-cluster-ip-range=10.96.0.0/12
```
Phân tích:

- `--etcd-servers=https://127.0.0.1:2379`: Chứng minh apiserver là component duy nhất connect tới database `etcd` qua giao thức HTTPS cục bộ
- `--authorization-mode=Node,RBAC`: Xác nhận apiserver sử dụng RBAC để phân quyền user và Node authorizer để phân quyền cho Kubelet
- `--service-cluster-ip-range=10.96.0.0/12`: Đây là nơi định nghĩa dải IP ảo cấp cho các Service (ClusterIP) trên toàn cụm

Schedule lấy quyền ở đâu để giao tiếp với API Server

Như chúng ta đã biết, để nói chuyện với API Server thì sẽ phải thông qua các lớp bảo mật. Scheduler là một component độc lập. Để gọi được API Server, nó phải có chứng chỉ xác thực (Kubeconfig)

```
root@k8s-master:~/.openclaw# cat /etc/kubernetes/manifests/kube-scheduler.yaml | grep kubeconfig
    - --authentication-kubeconfig=/etc/kubernetes/scheduler.conf
    - --authorization-kubeconfig=/etc/kubernetes/scheduler.conf
    - --kubeconfig=/etc/kubernetes/scheduler.conf
      name: kubeconfig
    name: kubeconfig
```
Kubelet đã mount file `/etc/kubernetes/scheduler.conf` từ host OS vào bên trong container của Scheduler để cấp quyền cho nó thực thi vòng lặp `filter/score`

### Inspect Kubelet Core Configuration
`kubelet` nhận lệnh từ `a`piserver`, tuy nhiên bản thân nó có một file cấu hình vật lý quyết định cách nó tương tác với Linux Kernel (systemd) và mạng (CoreDNS)

```
root@k8s-worker-2:~# cat /var/lib/kubelet/config.yaml | grep -E "cgroupDriver|clusterDNS|clusterDomain"
cgroupDriver: systemd
clusterDNS: [10.96.0.10]
clusterDomain: cluster.local
root@k8s-worker-2:~#
```
Phân tích:
- `cgroupDriver: systemd`: Kubelet được config để đồng bộ với bộ quản lý resource của Ubuntu (systemd) thay vì tự dùng cgroupfs, tránh xung đột kernel panic
- `clusterDNS: [10.96.0.10]`: Kubelet tự động inject IP này vào file `/etc/resolv.conf` của mọi Pod khởi tạo trên node này, ép các Pod phải resolve domain thông qua CoreDNS của K8s

### Kernel Rules của Kube-proxy
Đây là phần low-level nhất. kube-proxy mặc định chạy ở mode iptables. Khi một Service được tạo trên K8s, kube-proxy sẽ dịch object đó thành các luật NAT (Network Address Translation) chèn thẳng vào bảng định tuyến của Linux

#### Chúng ta sẽ dump toàn bộ rules mà kube-proxy đã thao túng

```
root@k8s-worker-2:~# sudo iptables -t nat -S | grep KUBE-SERVICES | head -n 10
-N KUBE-SERVICES
-A PREROUTING -m comment --comment "kubernetes service portals" -j KUBE-SERVICES
-A OUTPUT -m comment --comment "kubernetes service portals" -j KUBE-SERVICES
-A KUBE-SERVICES -d 10.96.0.10/32 -p udp -m comment --comment "kube-system/kube-dns:dns cluster IP" -m udp --dport 53 -j KUBE-SVC-TCOU7JCQXEZGVUNU
-A KUBE-SERVICES -d 10.96.0.10/32 -p tcp -m comment --comment "kube-system/kube-dns:dns-tcp cluster IP" -m tcp --dport 53 -j KUBE-SVC-ERIFXISQEP7F7OF4
-A KUBE-SERVICES -d 10.96.0.10/32 -p tcp -m comment --comment "kube-system/kube-dns:metrics cluster IP" -m tcp --dport 9153 -j KUBE-SVC-JD5MR3NA4I4DYORP
-A KUBE-SERVICES -d 10.96.0.1/32 -p tcp -m comment --comment "default/kubernetes:https cluster IP" -m tcp --dport 443 -j KUBE-SVC-NPX46M4PTMTKRN6Y
-A KUBE-SERVICES -m comment --comment "kubernetes service nodeports; NOTE: this must be the last rule in this chain" -m addrtype --dst-type LOCAL -j KUBE-NODEPORTS
```
Toàn bộ traffic (tcp/udp) đi qua máy Worker này đều bị Linux Kernel chặn lại ở rule có cờ -j KUBE-SERVICES. Nó bẻ lái traffic vào một danh sách các luật riêng do K8s kiểm soát.

#### Luồng đi của một Service cụ thể (VD: CoreDNS)

K8s luôn có sẵn một service tên là kube-dns với IP tĩnh 10.96.0.10 (như đã thấy ở config kubelet). Hãy tìm rule chính xác định tuyến cho IP này:

```
root@k8s-worker-2:~# sudo iptables -t nat -L KUBE-SERVICES -n | grep 10.96.0.10
KUBE-SVC-TCOU7JCQXEZGVUNU  17   --  0.0.0.0/0            10.96.0.10           /* kube-system/kube-dns:dns cluster IP */ udp dpt:53
KUBE-SVC-ERIFXISQEP7F7OF4  6    --  0.0.0.0/0            10.96.0.10           /* kube-system/kube-dns:dns-tcp cluster IP */ tcp dpt:53
KUBE-SVC-JD5MR3NA4I4DYORP  6    --  0.0.0.0/0            10.96.0.10           /* kube-system/kube-dns:metrics cluster IP */ tcp dpt:9153
```
Bạn sẽ thấy một rule có dạng: `KUBE-SVC-xxxxxx  tcp  --  0.0.0.0/0   10.96.0.10  tcp dpt:53`
Nghĩa là nếu có bất kỳ traffic nào đập vào IP `10.96.0.10` tại port 53, Linux Kernel sẽ không drop, mà nó jump (nhảy) sang một chain khác có tên là `KUBE-SVC-xxxxxx`

#### Xem rule DNAT (Đích đến cuối cùng - Pod IP)

kube-proxy tiếp tục tạo các chain KUBE-SEP-xxxx (Service Endpoint)
```
root@k8s-worker-2:~# sudo iptables -t nat -S | grep KUBE-SVC-TCOU7JCQXEZGVUNU
-N KUBE-SVC-TCOU7JCQXEZGVUNU
-A KUBE-SERVICES -d 10.96.0.10/32 -p udp -m comment --comment "kube-system/kube-dns:dns cluster IP" -m udp --dport 53 -j KUBE-SVC-TCOU7JCQXEZGVUNU
-A KUBE-SVC-TCOU7JCQXEZGVUNU ! -s 10.244.0.0/16 -d 10.96.0.10/32 -p udp -m comment --comment "kube-system/kube-dns:dns cluster IP" -m udp --dport 53 -j KUBE-MARK-MASQ
-A KUBE-SVC-TCOU7JCQXEZGVUNU -m comment --comment "kube-system/kube-dns:dns -> 10.244.2.2:53" -m statistic --mode random --probability 0.50000000000 -j KUBE-SEP-TCIZBYBD3WWXNWF5
-A KUBE-SVC-TCOU7JCQXEZGVUNU -m comment --comment "kube-system/kube-dns:dns -> 10.244.2.3:53" -j KUBE-SEP-ZHICQ2ODADGCY7DS
root@k8s-worker-2:~#
```
Bạn sẽ thấy rule này chia tỉ lệ phần trăm traffic (sử dụng module statistic mode random probability) để phân bổ tải đều sang các Pod CoreDNS backend. Sau đó nó thực thi DNAT (Destination NAT) chuyển đổi IP ảo `10.96.0.10` thành IP thật của Pod

Không có bất kỳ phép thuật hay thiết bị mạng đắt tiền nào ở đây cả. Chỉ có Netfilter và Iptables của Linux Kernel. Hãy cùng mổ xẻ từng dòng code mà kube-proxy đã tiêm vào hệ điều hành của bạn:

##### 1. The Entrypoint (Điểm chặn bắt gói tin)
```
-A PREROUTING -m comment --comment "kubernetes service portals" -j KUBE-SERVICES
-A OUTPUT -m comment --comment "kubernetes service portals" -j KUBE-SERVICES
```
Bản chất: `kube-proxy` đã hook (móc) vào 2 chuỗi gốc của Linux là PREROUTING (traffic từ ngoài node đi vào) và OUTPUT (traffic do chính tiến trình trên node tạo ra).

Hành động: Bất kỳ gói tin nào phát sinh, nó lập tức bị ném (-j) sang chain KUBE-SERVICES để kiểm tra.

##### 2. The Service Routing (Định tuyến tới CoreDNS)
```
-A KUBE-SERVICES -d 10.96.0.10/32 -p udp -m udp --dport 53 -j KUBE-SVC-TCOU7JCQXEZGVUNU
```
Bản chất: Khi một Pod khác trên máy Worker U82 cần phân giải tên miền (VD: ping google.com), nó sẽ gửi gói tin UDP đến IP tĩnh của Service CoreDNS là `10.96.0.10` tại port 53

Hành động: Linux Kernel bắt được gói tin có đích đến là `10.96.0.10:53`. Nó không drop, mà bẻ lái (-j) sang chain chuyên biệt của Service đó: `KUBE-SVC-TCOU7JCQXEZGVUNU`

##### 3. L4 Load Balancing bằng thuật toán Xác suất (The Core Magic)
```
-A KUBE-SVC-TCOU7JCQXEZGVUNU -m comment --comment "kube-system/kube-dns:dns -> 10.244.2.2:53" -m statistic --mode random --probability 0.50000000000 -j KUBE-SEP-TCIZBYBD3WWXNWF5
-A KUBE-SVC-TCOU7JCQXEZGVUNU -m comment --comment "kube-system/kube-dns:dns -> 10.244.2.3:53" -j KUBE-SEP-ZHICQ2ODADGCY7DS
```
Bản chất: kube-proxy phát hiện Service CoreDNS này đang có 2 Pod backend đang chạy (IP là `10.244.2.2` và `10.244.2.3`). Nó phải chia đều tải 50/50. Nhưng iptables đọc luật từ trên xuống dưới, làm sao chia tải? Nó dùng module xt_statistic của Linux Kernel.

Giải mã Logic:
- Rule 1: Gói tin đi vào có đúng 50% xác suất (probability 0.5) khớp với luật này. Nếu khớp, nó nhảy sang chain `KUBE-SEP-TCIZBYBD3WWXNWF5` (để thực hiện DNAT đổi đích thành 10.244.2.2).
- Rule 2: Nếu gói tin "trượt" xác suất 50% ở rule 1, nó trôi xuống rule 2. Ở đây không còn điều kiện xác suất nào nữa (xác suất 100%), gói tin chắc chắn bị ném sang chain `KUBE-SEP-ZHICQ2ODADGCY7DS` (DNAT thành 10.244.2.3).
