# API SERVER VÀ ETCD
Máy `172.16.66.84` đã cài kubeadm, kubelet, nhưng chúng đang "ngủ". Chúng ta sẽ dùng `kubeadm` để khởi động Control Plane.

## LÝ THUYẾT CHUYÊN SÂU
Khi nói về "Control Plane", chúng ta đang nói về một tập hợp các tiến trình chạy ngầm. Trong đó, API Server và etcd là cặp bài trùng cốt lõi nhất.

### 1. etcd: Trí nhớ của Kubernetes 
Bản chất: etcd không phải là một cơ sở dữ liệu quan hệ (SQL) cồng kềnh. Nó là một kho lưu trữ dạng Key-Value (Khóa - Giá trị) phân tán, có tính nhất quán cực cao dựa trên thuật toán Raft.

**Nhiệm vụ**: Nó lưu trữ MỌI THỨ của cụm K8s. Từ cấu hình mạng, danh sách các Node, cho đến việc Pod Nginx đang chạy ở IP nào,...

Nguyên tắc: Trong K8s, etcd là nguồn chân lý duy nhất (Single Source of Truth). Nếu etcd báo rằng hệ thống cần có 3 Pod Nginx, thì toàn bộ cụm sẽ làm mọi cách để tạo ra đúng 3 Pod đó. Nếu etcd bị xóa sạch dữ liệu, cụm K8s sẽ "mất trí nhớ" và sụp đổ hoàn toàn.

Định dạng dữ liệu: Để tối ưu tốc độ và dung lượng, K8s không lưu dữ liệu trong etcd dưới dạng JSON thông thường. Nó serialize (tuần tự hóa) dữ liệu thành định dạng Protobuf (Protocol Buffers của Google)

**Bảo mật**: Không một thành phần nào trong K8s (kể cả Scheduler hay Kubelet) được phép nói chuyện trực tiếp với etcd. Ngoại lệ duy nhất là API Server.

### 2. kube-apiserver: Cổng giao tiếp độc quyền
Bản chất: Đây là một RESTful API Server. Mọi tương tác trong cụm K8s (từ bạn gõ lệnh kubectl, từ Worker Node báo cáo trạng thái) đều là các lệnh gọi HTTP (GET, POST, PUT, DELETE) gửi đến API Server.

Quy trình 3 bước bảo vệ của API Server: Khi bạn gửi một lệnh yêu cầu tạo Pod, API Server không làm ngay mà qua 3 trạm kiểm lâm:

![](/Anh/Screenshot_954.png)
- **Authentication** (Xác thực): Bạn là ai? Có chứng chỉ hợp lệ không?
- **Authorization** (Phân quyền - RBAC): Bạn là user hợp lệ, nhưng bạn có quyền "Tạo Pod" ở namespace này không?
- **Admission Control** (Kiểm soát nhập học): Lệnh hợp lệ, quyền hợp lệ, nhưng Pod này có vi phạm luật của hệ thống không? (Ví dụ: Pod này có đòi hỏi dùng quyền Root của máy chủ Linux không? Nếu có, từ chối).

Sau khi qua 3 vòng này, API Server mới ghi bản ghi "Cần tạo 1 Pod" vào etcd. Xong việc của nó!

## THỰC HÀNH KHỞI TẠO 

### Bước 1: Khởi động API Server và etcd
Mở terminal trên máy `172.16.66.84` và chạy lệnh sau:
```
kubeadm init --control-plane-endpoint="172.16.66.84" --pod-network-cidr=10.244.0.0/16
```
Điều gì đang thực sự xảy ra dưới nền khi bạn chạy lệnh này?
- Sinh chứng chỉ (Certificates): Kubeadm đang tạo ra một loạt chứng chỉ SSL mã hóa (Lưu ở /etc/kubernetes/pki/). API Server dùng chứng chỉ này để mã hóa kết nối HTTPS.
- Tạo Kubeconfig: Tạo file cấu hình chứa quyền admin.
- Tạo Static Pods: Kubeadm tạo ra các file định nghĩa `.yaml` và vứt vào thư mục `/etc/kubernetes/manifests/`. Kubelet sẽ thấy các file này và tự động ra lệnh cho Containerd chạy API Server và etcd dưới dạng các container (gọi là Static Pods).

### Bước 2: Nhận quyền điều khiển (Cấu hình Kubectl)
Để API Server biết bạn là Admin (vượt qua vòng Authentication), bạn cần copy file chứng chỉ mà kubeadm vừa tạo ra vào thư mục user của bạn:
```
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config
```

### Dữ liệu của etcd nằm ở đâu trên ổ cứng?
Dù chạy bằng container, nhưng etcd phải lưu dữ liệu xuống ổ cứng thật để tắt máy không bị mất. Kubeadm mặc định cấu hình thư mục lưu trữ tại /var/lib/etcd. Hãy thử kiểm tra:
```
sudo ls -l /var/lib/etcd/member/snap/
```
Đó chính là nơi chứa các file cơ sở dữ liệu định hình nên toàn bộ cụm K8s của bạn.
```
root@MANH-U84:~#sudo ls -l /var/lib/etcd/member/snap//
total 1464
-rw------- 1 root root 16801792 Mar 23 16:42 db
```
### Vậy trong etcd lưu cái gì?
etcd được K8s bảo vệ cực kỳ nghiêm ngặt bằng mTLS (chứng chỉ hai chiều). Bạn không thể cài một tool database bên ngoài để soi vào được. Chúng ta phải chui thẳng vào container của etcd và dùng công cụ etcdctl của chính nó.

Xác định tên Pod của etcd:
```
kubectl get pods -n kube-system | grep etcd
```
```
etcd-k8s-master                      1/1     Running   0          65m
```
Chui vào bên trong
```
kubectl exec -it -n kube-system etcd-k8s-master -- sh
```
Thiết lập công cụ `etcdctl`
```
export ETCDCTL_API=3
alias etcdctl='etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key'
```
Khám phá cấu trúc thư mục của K8s trong etcd:
Hãy xem K8s tổ chức dữ liệu thành các "đường dẫn" (keys) như thế nào.
```
etcdctl get /registry/ --prefix --keys-only
```
Kết quả: Bạn sẽ thấy một danh sách dài dằng dặc. Mọi thứ đều nằm dưới key /registry/.
Ví dụ:
```
/registry/namespaces/default (Lưu thông tin namespace)

/registry/pods/kube-system/kube-apiserver-k8s-master (Lưu thông tin Pod)

/registry/minions/k8s-master (Lưu thông tin Node - K8s ngày xưa gọi Node là minion).
```
Đọc dữ liệu thô của một Node (Máy Master của bạn):
```
etcdctl get /registry/minions/k8s-master
```
![](/Anh/Screenshot_1047.png)
Phân tích kết quả: Bạn sẽ thấy một mớ ký tự lộn xộn, xen lẫn vài từ tiếng Anh đọc được (như tên node, địa chỉ IP). Đây chính là định dạng Protobuf đã được tối ưu hóa dung lượng. K8s không lưu JSON thô vào đây để tiết kiệm RAM tối đa. Chỉ có API Server mới có bộ dịch (codec) để chuyển cục Protobuf này ngược lại thành JSON cho bạn xem.

Thoát khỏi etcd container:
```
exit
```
