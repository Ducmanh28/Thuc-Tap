# Masternode trong K8S
## 1. Khái Niệm Tổng Quan
Control Plane là tập hợp các tiến trình (processes) quản lý trạng thái tổng thể của toàn bộ cụm K8s. Nó đóng vai trò đưa ra các quyết định ở tầm vĩ mô (như lập lịch chạy Pod), phát hiện và phản hồi lại các sự kiện của hệ thống (ví dụ: khởi tạo Pod mới khi một ReplicaSet không đủ số lượng).

Trong mô hình production chuẩn, Control Plane thường được chạy trên nhiều máy chủ (High Availability - HA) để tránh điểm nghẽn đơn lẻ (Single Point of Failure). Với lab 3 node của chúng ta, các thành phần này tập trung trên 1 Master Node.

## 2. Các Thành Phần 

![](/Anh/Screenshot_1061.png)

Control Plane được cấu thành từ 4 thành phần chính, hoạt động phối hợp chặt chẽ với nhau:

- `kube-apiserver`: Điểm entrypoint duy nhất cho mọi request REST API. Cả user (qua kubectl), các thành phần khác của Control Plane, và Worker Node (qua kubelet) đều phải giao tiếp qua đây. Nó chịu trách nhiệm xác thực (Authentication), cấp quyền (Authorization) và kiểm tra tính hợp lệ (Admission Control) của dữ liệu trước khi lưu xuống database.

- `etcd`: Cơ sở dữ liệu dạng Key-Value lưu trữ toàn bộ trạng thái (State) và cấu hình của cluster. Nó đảm bảo tính nhất quán cao và là "nguồn chân lý" (Source of Truth) duy nhất. Nếu `etcd` mất dữ liệu, toàn bộ cụm K8s sẽ tê liệt.

- `kube-scheduler`: Lắng nghe các Pod mới được tạo nhưng chưa được gán (assign) cho node nào. Nó sẽ tính toán các rào cản tài nguyên (CPU/RAM), policy (Node Affinity, Taints/Tolerations) để chọn ra Worker Node tối ưu nhất cho Pod đó chạy.

- `kube-controller-manager`: Chạy các tiến trình điều khiển (control loops) chạy ngầm để liên tục so sánh trạng thái hiện tại (Current State) và trạng thái mong muốn (Desired State). Ví dụ: Node Controller theo dõi node chết, ReplicaSet Controller đảm bảo số lượng Pod luôn đúng như khai báo.

## 4. Kiểm tra từng thành phần
### 4.1. etcd
Chúng ta sử dụng `describe` để mô tả thông tin của 1 thành phần. Tuy nhiên nếu chúng ta dùng luôn:
```
root@k8s-master:~# kubectl describe etcd-k8s-master
error: the server doesn't have a resource type "etcd-k8s-master"
```
Sẽ ngay lập tức gặp lỗi vì server không biết namespace chứa `etcd-k8s-master` ở đâu.

Chúng ta sẽ dùng:
```
root@k8s-master:~# kubectl get all -A
NAMESPACE      NAME                                     READY   STATUS    RESTARTS   AGE
kube-system    pod/etcd-k8s-master                      1/1     Running   0          6d1h
kube-system    pod/kube-apiserver-k8s-master            1/1     Running   0          6d1h
kube-system    pod/kube-controller-manager-k8s-master   1/1     Running   0          6d1h
kube-system    pod/kube-scheduler-k8s-master            1/1     Running   0          6d1h
```
Vậy là bây giờ chúng ta đã có tên pod, namespace chứa pod

Mô tả etcd:
```
root@k8s-master:~# kubectl describe pod/etcd-k8s-master -n kube-system
```
#### 1. Định danh
Ngay từ những dòng đầu tiên, K8S đã cung cấp cho chúng ta các "định vị" quan trọng:

**Node**: `k8s-master/172.16.66.84`: Pod này đang chạy cứng trên node k8s-master có IP 172.16.66.84.

**Labels**: component=etcd, tier=control-plane: Các nhãn này giúp hệ thống (và cả chúng ta khi dùng lệnh kubectl get -l) dễ dàng phân loại đây là một thành phần thuộc Control Plane.

**Controlled By**: Node/k8s-master: Đây là "bằng chứng thép" (smoking gun) xác nhận đây là một Static Pod. Thay vì được quản lý bởi một ReplicaSet hay DaemonSet như các Pod ứng dụng thông thường, Pod này do chính tiến trình Kubelet trên máy chủ k8s-master trực tiếp quản lý và giữ cho nó sống.

#### 2. Cấu hình khởi chạy 
Phần giá trị nhất của log này nằm ở block Command:. Đây chính là các tham số (flags) được truyền vào binary của etcd khi khởi động. Kỹ sư hệ thống nhìn vào đây để bắt bệnh:

**Dữ liệu lõi**: `--data-dir=/var/lib/etcd` -> Chỉ định nơi etcd sẽ ghi database Key-Value của toàn cụm K8S.

Cấu hình Mạng (Networking):

- `--listen-client-urls=https://127.0.0.1:2379,https://172.16.66.84:2379`: etcd mở port 2379 để lắng nghe các request từ client (ở đây chính là kube-apiserver). Nó cho phép kết nối qua localhost và IP của Master node.

- `--listen-peer-urls=https://172.16.66.84:2380`: Mở port 2380 để giao tiếp với các node etcd khác. (Dù cụm của bạn hiện tại chỉ có 1 Master node, nó vẫn mở port này để sẵn sàng cho mô hình HA - High Availability ghép thêm Master sau này).

Bảo mật khắt khe (TLS Everywhere):

Hàng loạt các tham số như `--cert-file`, `--key-file`, `--trusted-ca-file`... cho thấy etcd không chấp nhận kết nối HTTP trần. Mọi giao tiếp với client (API Server) hay peer đều bị bắt buộc mã hóa qua TLS/SSL.

#### 3. Lưu trữ vật lý
Liên kết phần Command ở trên với block Volumes và Mounts ở cuối log, ta thấy rõ cách K8S giải quyết bài toán lưu trữ cho Static Pod:

Bảo vệ Dữ liệu:
```
etcd-data:
  Type: HostPath (bare host directory volume)
  Path: /var/lib/etcd
```
Dữ liệu của container được ánh xạ (Mounts) thẳng ra thư mục `/var/lib/etcd` trên ổ cứng vật lý của Master node. Nhờ HostPath, nếu container etcd này bị crash và restart lại, dữ liệu của K8S không hề bị mất đi.

Cung cấp Chứng chỉ:

Tương tự, các file chứng chỉ (.crt, .key) được mount từ thư mục `/etc/kubernetes/pki/etcd` của máy chủ vật lý vào trong container để nó có thể đọc được cấu hình bảo mật.

#### 4. Cơ chế Tự kiểm tra sức khỏe
Liveness & Startup: Hệ thống sử dụng HTTP GET tới địa chỉ `http://127.0.0.1:2381/health...` để liên tục "khám sức khỏe" cho etcd. Nếu endpoint này trả về lỗi (sau 8 lần thất bại với Liveness), Kubelet sẽ tự động "bắn hạ" container lỗi và khởi tạo lại container mới để tự phục hồi

Events: <none>: Pod này đang chạy cực kỳ ổn định, không có vòng lặp lỗi hay cảnh báo thiếu tài nguyên nào diễn ra gần đây

#### 5. Etcd lưu trữ cái gì?
etcd được K8s bảo vệ cực kỳ nghiêm ngặt bằng mTLS (chứng chỉ hai chiều). Bạn không thể cài một tool database bên ngoài để soi vào được. Chúng ta phải chui thẳng vào container của etcd và dùng công cụ etcdctl của chính nó.

Xác định tên Pod của etcd:
```
kubectl get pods -n kube-system | grep etcd
etcd-k8s-master                      1/1     Running   0          65m
```

Chui vào bên trong
```
kubectl exec -it -n kube-system etcd-k8s-master -- sh
```
Thiết lập công cụ `etcdctl`
```
export ETCDCTL_API=3
alias etcdctl='etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key'
```
Khám phá cấu trúc thư mục của K8s trong etcd:
Hãy xem K8s tổ chức dữ liệu thành các "đường dẫn" (keys) như thế nào.
```
etcdctl get /registry/ --prefix --keys-only
```
Kết quả: Bạn sẽ thấy một danh sách dài dằng dặc. Mọi thứ đều nằm dưới key /registry/.
Ví dụ:
```
/registry/namespaces/default (Lưu thông tin namespace)

/registry/pods/kube-system/kube-apiserver-k8s-master (Lưu thông tin Pod)

/registry/minions/k8s-master (Lưu thông tin Node - K8s ngày xưa gọi Node là minion).
```
Đọc dữ liệu thô của một Node:
```
etcdctl get /registry/minions/k8s-master
```
![](/Anh/Screenshot_1047.png)

Phân tích kết quả: Bạn sẽ thấy một mớ ký tự lộn xộn, xen lẫn vài từ tiếng Anh đọc được (như tên node, địa chỉ IP). Đây chính là định dạng Protobuf đã được tối ưu hóa dung lượng. K8s không lưu JSON thô vào đây để tiết kiệm RAM tối đa. Chỉ có API Server mới có bộ dịch (codec) để chuyển cục Protobuf này ngược lại thành JSON cho bạn xem.

Thoát khỏi etcd container:
```
exit
```

### 4.2 api-server
Mô tả:
```
root@k8s-master:~# kubectl describe pod/kube-apiserver-k8s-master -n kube-system
```
#### 1. Định danh và Bản chất 
**Controlled By**: `Node/k8s-master`: Tương tự như etcd, kube-apiserver cũng là một Static Pod do kubelet trên Master node trực tiếp quản lý

**Image**: `registry.k8s.io/kube-apiserver:v1.29.15`: Cụm K8S của bạn đang chạy phiên bản v1.29.15 

#### 2. Mạng và Giao tiếp 
Khối Command cho chúng ta thấy rõ cách API Server giao tiếp với "thế giới":

`--secure-port=6443`: API Server đang mở port chuẩn 6443. Mọi request gửi tới cluster đều phải nhắm vào port này.

`--advertise-address=172.16.66.84`: Nó công bố địa chỉ IP `172.16.66.84` (IP của Master node) cho các thành phần khác trong cluster biết để gọi tới.

`--etcd-servers=https://127.0.0.1:2379`: API Server đang gọi trực tiếp tới etcd qua IP localhost ở port 2379 để đọc/ghi dữ liệu.

`--service-cluster-ip-range=10.96.0.0/12`: Một thông số mạng cực kỳ quan trọng. Nó định nghĩa dải IP ảo sẽ được cấp phát riêng cho các Kubernetes Services (ClusterIP)

#### 3. Hàng rào Bảo mật 
kube-apiserver là thành phần bị khóa chặt nhất trong K8S. Bạn có thể thấy cấu hình của nó tràn ngập các policy và chứng chỉ:

- `--authorization-mode=Node,RBAC`: Hệ thống phân quyền nhiều lớp. K8S sẽ kiểm tra quyền dựa trên Node và RBAC (Role-Based Access Control - phân quyền chi tiết cho từng user/service account).

- `--enable-admission-plugins=NodeRestriction`: Admission Controller là chốt chặn cuối cùng trước khi ghi vào database. NodeRestriction ngăn không cho một Worker Node tự ý sửa đổi cấu hình của các Node khác, chống rò rỉ đặc quyền.

Các tham số `*cert-file` và `*key-file`: Có tới hàng chục đường dẫn chứng chỉ (ví dụ: `--tls-cert-file`, `--kubelet-client-certificate`, `--etcd-certfile`). Điều này minh chứng rằng API Server đóng vai trò trung tâm xác thực mTLS (Mutual TLS). Nó vừa làm Server (yêu cầu client trình chứng chỉ), vừa làm Client (trình chứng chỉ khi gọi sang etcd hoặc kubelet).

#### 4. Volumes & Mounts
Trái ngược với etcd cần ghi dữ liệu, kube-apiserver là một tiến trình Stateless (không lưu trạng thái nội bộ).

phân tích Volume: Nhìn vào khối Mounts. Tất cả các mapping từ HostPath vào container (như `/etc/kubernetes/pki`, `/etc/ssl/certs`) đều có chữ (ro) ở cuối

Kết luận : API Server chỉ được phép đọc các file chứng chỉ bảo mật từ hệ điều hành Master Node mà không có quyền sửa đổi hay xóa chúng. Đây là một cấu hình Hardening chuẩn mực của hệ thống

#### 5. Cơ chế Khám sức khỏe (Health Probes)
Không chỉ kiểm tra trạng thái sống còn , API Server còn có endpoint `/readyz`. Kubelet sẽ liên tục hit vào `https://172.16.66.84:6443/readyz`. Chỉ khi endpoint này trả về HTTP 200, API Server mới được coi là "sẵn sàng" nhận traffic, tránh trường hợp hệ thống định tuyến nhầm request vào API Server khi nó đang khởi động dang dở.

### 4.3 Control manager
Mô tả:
```
root@k8s-master:~# kubectl describe pod/kube-controller-manager-k8s-master -n kube-system
```
#### 1. Deployment Model & Identity
**Controlled By**: `Node/k8s-master`: Trạng thái này xác nhận `kube-controller-manager` được deploy dưới dạng Static Pod. Tiến trình kubelet trên node `172.16.66.84` trực tiếp đọc manifest từ thư mục cục bộ (mặc định `/etc/kubernetes/manifests/`) để khởi tạo và quản lý vòng đời container, bypass hoàn toàn kube-apiserver và kube-scheduler

**Image**: Chạy version `v1.29.15` qua `container runtime containerd`

#### 2. Networking & API Communication
`--bind-address=127.0.0.1`: Tiến trình chỉ binding port trên interface loopback. Nó không expose bất kỳ port nào ra interface vật lý hay cluster network

`--kubeconfig=/etc/kubernetes/controller-manager.conf`: Thành phần này hoạt động dưới vai trò là một REST Client. Nó sử dụng file kubeconfig để thực hiện Authentication (qua client-certificate) và Authorization khi gọi các API endpoints của kube-apiserver

**Probes**: Liveness và Startup probes thực hiện HTTP GET healthcheck qua port 10257 trên localhost

#### 3. Phân tích tham số thực thi (Command Flags)
Khối Command định nghĩa các tham số kích hoạt các Control Loops (tiến trình background) cốt lõi:

Quản lý IPAM (IP Address Management):

- `--allocate-node-cidrs=true` và `--cluster-cidr=10.244.0.0/16`: Kích hoạt Node Controller cấp phát các subnet nhỏ hơn (thường là /24) từ dải mạng overlay `10.244.0.0/16` cho từng Worker Node khi chúng join cluster. Thông số này là cơ sở để CNI Plugin (ví dụ: Flannel, Calico) thiết lập routing table cho Pod network.

Quản lý ServiceAccount (Cryptographic Signing):

`--service-account-private-key-file=/etc/kubernetes/pki/sa.key`: Kích hoạt ServiceAccount Controller. Nó sử dụng RSA private key (sa.key) để ký (sign) các JSON Web Tokens (JWT) cấp phát cho Pods khi chúng cần giao tiếp với K8s API.

Controllers mặc định & TLS Bootstrapping:

`--controllers=*,bootstrapsigner,tokencleaner`: Chạy toàn bộ các controller mặc định (ReplicaSet, DaemonSet, Endpoint...). Cờ bootstrapsigner và tokencleaner đặc biệt quan trọng để vận hành cơ chế Kubelet TLS Bootstrapping (tự động cấp và xoá chứng chỉ CSR cho Worker Nodes).

#### 4. Cơ chế High Availability (Leader Election)
`--leader-elect=true`: Kích hoạt thuật toán Leader Election bằng Distributed Locks (thông qua resource Leases trong API Server). Trong kiến trúc HA Control Plane (Multi-Master), cơ chế này đảm bảo tại một thời điểm, chỉ có duy nhất 1 instance kube-controller-manager nắm giữ lock và ở trạng thái Active (thực thi control loops). Các instances khác ở trạng thái Standby để tránh tình trạng Race Condition ghi đè trạng thái hệ thống.

#### 5. Storage Mapping & Stateless Architecture
Mounts (ro): Khối Volumes và Mounts cho thấy tất cả các directories chứa Public Key Infrastructure (PKI) như `/etc/kubernetes/pki` và file `controller-manager.conf` được mount từ Host filesystem vào container với quyền Read-Only (ro).

### 4.4 Scheduler
Mô tả:
```
root@k8s-master:~# kubectl describe pod/kube-scheduler-k8s-master -n kube-system
```
#### Deployment Model & Identity
**Controlled By**: `Node/k8s-master`: Tương tự API Server và Controller Manager, kube-scheduler được triển khai dưới dạng Static Pod. Tiến trình kubelet bypass K8s API, trực tiếp mount và khởi tạo container từ file manifest `/etc/kubernetes/manifests/kube-scheduler.yaml` trên node cục bộ.

**Image**: Sử dụng image `registry.k8s.io/kube-scheduler:v1.29.15` thông qua container runtime containerd.

**Priority Class**: `system-node-critical`: Đảm bảo Pod này có độ ưu tiên cao nhất về mặt tài nguyên trên node, không bị evict khi hệ thống gặp áp lực memory/CPU.

#### Networking & API Communication
`--bind-address=127.0.0.1`: Tiến trình thực hiện binding socket trực tiếp trên interface loopback (127.0.0.1). Không expose bất kỳ port nào ra external network hoặc cluster overlay network để giảm thiểu attack surface.

**Health Probes**: Liveness và Startup probe thực hiện tuần tự các HTTP GET request tới endpoint bảo mật `https://127.0.0.1:10259/healthz`. Nếu fail vượt ngưỡng (threshold), kubelet sẽ trigger container restart.

Authentication/Authorization:

- `--authentication-kubeconfig` và `--authorization-kubeconfig`: Xác định cách Scheduler chứng thực chính nó với các request nội bộ.

- `--kubeconfig=/etc/kubernetes/scheduler.conf`: Scheduler hoạt động như một REST Client. Nó sử dụng file kubeconfig này (chứa client certificate) để thiết lập kết nối mTLS tới kube-apiserver nhằm theo dõi (watch) các Pod có trạng thái nodeName: "" (chưa được schedule).

#### Cơ chế High Availability 
`--leader-elect=true`: Kích hoạt cơ chế Leader Election thông qua K8s API

**Mục đích cấu trúc**: Trong topology Multi-Master (HA Control Plane), nhiều instance kube-scheduler sẽ chạy đồng thời. Cơ chế distributed lock này đảm bảo chỉ có 1 instance ở trạng thái Active (thực hiện thuật toán Filtering và Scoring để gán Pod vào Node), các instance còn lại duy trì trạng thái Standby để tránh xung đột dữ liệu (Race condition).

#### Storage Mapping & Stateless Architecture
**Mounts**: Khối Volume cấu hình HostPath ánh xạ file `/etc/kubernetes/scheduler.conf` từ file system của Master Node vào container với quyền Read-Only (ro).

**Kiến trúc Stateless**: Khối cấu hình này chứng minh kube-scheduler hoàn toàn Stateless. Tiến trình không lưu trữ bất kỳ trạng thái cục bộ nào. Mọi context về Node capacity, resource request/limit của Pod đều được query trực tiếp từ etcd (qua API Server) trong thời gian thực để phục vụ thuật toán scheduling.
