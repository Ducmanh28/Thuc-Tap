# WorkerNode trong K8S
## Kiến Trúc Và Các Thành Phần Cốt Lõi

![](/Anh/Screenshot_1061.png)

Khác với Master Node thường chạy các thành phần dưới dạng Static Pods, Worker Node có sự pha trộn giữa các tiến trình (services) chạy trực tiếp trên hệ điều hành (OS-level) và các Pods được quản lý bởi hệ thống (DaemonSets).

**kubelet (Node Agent)**:

Chức năng: Tiến trình daemon cốt lõi chạy trên mọi Node. kubelet nhận các đặc tả Pod từ kube-apiserver và tương tác với Container Runtime qua giao thức CRI (Container Runtime Interface) để đảm bảo các container được khởi tạo và duy trì trạng thái Running. Nó cũng chịu trách nhiệm báo cáo Node Status (CPU/Memory usage, disk space) và Pod Status ngược về API Server.

**Container Runtime (CRI Implementation)**:

Chức năng: Phần mềm thực thi container ở cấp độ kernel Linux (ví dụ: containerd, CRI-O). Nó chịu trách nhiệm pulling images từ registry, unpack filesystem, và thiết lập cgroups/namespaces cho container.

**kube-proxy (Network Proxy)**:

Chức năng: Thành phần định tuyến mạng cục bộ trên mỗi Node, thực thi khái niệm Kubernetes Service. kube-proxy theo dõi API Server để lấy thông tin các object Service và Endpoint, từ đó tự động biên dịch thành các rules định tuyến ở tầng network kernel (thông qua iptables hoặc IPVS), đảm bảo traffic được forward chính xác tới các backend Pods.


## Vị Trí Triển Khai 
Tiến trình mức Hệ điều hành (Systemd Services):

kubelet và containerd (hoặc runtime khác) KHÔNG chạy dưới dạng Pod. Chúng được quản lý bởi systemd trên Linux.

Cấu hình của kubelet thường nằm tại: `/var/lib/kubelet/config.yam`l và sử dụng `/etc/kubernetes/kubelet.conf `(kubeconfig) để xác thực TLS với API Server.

Tiến trình mức Cluster (DaemonSet):

kube-proxy và CNI Plugin (Container Network Interface như Flannel, Calico) được triển khai dưới dạng DaemonSet. Điều này có nghĩa là K8s scheduler tự động đảm bảo mỗi Worker Node khi join vào cluster sẽ luôn có sẵn 1 bản sao Pod của kube-proxy và CNI chạy trên đó.

### Định danh và Topology (Identity & Topology)
Dữ liệu thực tế: `Roles: worker, Labels: node-role.kubernetes.io/worker=worker`.

Node đã được cập nhật nhãn (label) định danh. Trong hệ thống Kubernetes phân tán, thành phần kube-scheduler không định tuyến dựa trên tên máy chủ (hostname) mà dựa trên các labels này. Việc gán label worker thiết lập cơ sở dữ liệu metadata để kỹ sư hệ thống có thể áp dụng các tham số như NodeSelector hoặc NodeAffinity vào file YAML của ứng dụng, buộc API Server phải lập lịch (schedule) các Application Pods về đúng cụm node xử lý, tách biệt với Control Plane.

### Kiến trúc Mạng Overlay (Overlay Networking)
Dữ liệu thực tế: `flannel.alpha.coreos.com/backend-type: vxlan, PodCIDR: 10.244.2.0/24`.

Node sử dụng CNI Plugin là Flannel với công nghệ đóng gói VXLAN (Virtual eXtensible Local Area Network). Điều này cho phép tạo ra một mạng L2 ảo (Overlay) chạy trên nền hạ tầng vật lý L3 (Underlay).

IPAM của cluster đã cấp phát độc quyền dải subnet `10.244.2.0/24` cho node này. Container Runtime (containerd) sẽ cấp phát IP cho mọi Pod mới khởi tạo trên node nằm trong dải này. Thiết kế chia subnet độc lập cho từng node loại bỏ triệt để xung đột địa chỉ IP (IP conflict) trong toàn bộ cụm.

### Quản lý Tài nguyên Cấp phát (Resource Allocation)
Dữ liệu thực tế:  
```
Capacity: CPU 2, Memory ~2GB

Allocatable: CPU 2, Memory ~1.9GB.

Pods: 110.
```

- Capacity hiển thị tổng lượng tài nguyên phần cứng vật lý mà Kernel nhận diện.

- Allocatable là lượng tài nguyên tối đa mà K8S cho phép cấp cho các Pods. Sự chênh lệch (khoảng 100MB RAM) chính là cơ chế System/Kube Reserved. kubelet chủ động khóa lượng RAM này lại để đảm bảo hệ điều hành cơ sở (Ubuntu) và các tiến trình nền tảng không bao giờ bị thiếu hụt bộ nhớ. Cơ chế này ngăn chặn trực tiếp rủi ro Node bị treo do lỗi OOM (Out-Of-Memory) từ cấp độ OS.

- Giới hạn 110 Pods là cấu hình mặc định được nội suy từ dải mạng /24 (có tối đa 254 IP khả dụng), đảm bảo tỷ lệ an toàn chống cạn kiệt IP cục bộ (IP Exhaustion).

### Đánh giá Sức khỏe và Chu kỳ Giám sát 
Dữ liệu thực tế: `Ready: True, MemoryPressure / DiskPressure / PIDPressure: False, Lease RenewTime: Sun, 29 Mar 2026 18:23:47`.


- Cơ chế Node Lease đang hoạt động ổn định. Tiến trình kubelet gửi thành công các tín hiệu keep-alive (heartbeats) định kỳ về kube-apiserver để duy trì phiên kết nối

- Các chỉ số áp lực tài nguyên (Pressure Metrics) giám sát sự cạn kiệt tài nguyên cục bộ. Vì các giá trị này đều là False, node chưa chạm ngưỡng tới hạn. Nếu bất kỳ chỉ số nào chuyển sang True, kubelet sẽ tự động kích hoạt tiến trình Pod Eviction (thu hồi tài nguyên bằng cách chủ động gửi tín hiệu SIGTERM để chấm dứt các Pods có độ ưu tiên thấp)

### Khối lượng Công việc Hiện tại (Current Workloads)
Dữ liệu thực tế: 4 Pods hệ thống đang chạy (kube-flannel, kube-proxy, 2 coredns). Tổng Requests: 15% CPU, 10% Memory.

Node đang duy trì các tiến trình DaemonSet phụ trách cấu hình routing (iptables/IPVS) và overlay network, đồng thời chịu trách nhiệm phân giải Service Discovery (coredns). 

Dựa trên tổng lượng tài nguyên cam kết (Requests), node đang ở trạng thái nhàn rỗi (idle resource cao). Hệ thống Kube-scheduler có thể lập lịch thêm số lượng lớn các Pods ứng dụng vào node này dựa trên lượng tài nguyên Allocatable còn trống.

Tham số Giao tiếp (Command-line arguments):

- Dòng ExecStart hiển thị rõ Kubelet dùng file `/etc/kubernetes/kubelet.conf` để xác thực với API Server.

- Cờ `--container-runtime-endpoint=unix:///var/run/containerd/containerd.sock` xác nhận nó đang móc nối trực tiếp với Containerd.

- Cờ `--pod-infra-container-image=registry.k8s.io/pause:3.9`: Đây là kiến thức chuyên sâu. Khi tạo bất kỳ Pod nào, Kubelet sẽ luôn gọi Containerd khởi tạo một container "tàng hình" mang tên pause phiên bản 3.9 trước tiên. Container này không chạy code, nó chỉ đứng đó để giữ chỗ cho Network Namespace, sau đó các container ứng dụng (như Nginx, CoreDNS) mới được nhét vào dùng chung mạng lưới đó.

#### Cấu trúc Phân tầng: Containerd-shim và Runc
Điểm đắt giá nhất trong bản log này là danh sách các tiến trình con trong CGroup. Nó minh chứng cho kiến trúc "Sidecar process" của containerd:

`containerd-shim-runc-v2`: Bạn thấy có 4 tiến trình shim đang chạy độc lập (PIDs: 25070, 56296, 56646, 56656).

Bản chất chuyên ngành: Mỗi Pod (hoặc đúng hơn là mỗi tập hợp container) sẽ có một tiến trình shim riêng biệt.

Tác dụng: shim đóng vai trò là "người trung gian" giữa containerd và runc. Nó cho phép containerd có thể restart hoặc nâng cấp mà không làm chết các container đang chạy bên dưới. Nó cũng chịu trách nhiệm báo cáo exit code của container về cho K8S.

`-namespace k8s.io`: Đây là tham số cực kỳ quan trọng. containerd hỗ trợ đa người dùng (multi-tenancy) thông qua namespace. Toàn bộ các container do Kubernetes quản lý bắt buộc phải nằm trong namespace k8s.io. Nếu bạn dùng Docker hoặc các công cụ khác trên cùng máy chủ này, chúng sẽ nằm ở các namespace khác (ví dụ: default), giúp cô lập hoàn toàn tài nguyên của Cluster.

#### Đối chiếu Workload (Mapping)
Dựa trên số lượng 4 tiến trình shim trong log, chúng ta có thể đối chiếu trực tiếp với danh sách Pods mà chúng ta đã thấy ở lệnh describe node trước đó:

- Shim 1: Phục vụ kube-flannel (Network).

- Shim 2: Phục vụ kube-proxy (Routing).

- Shim 3: Phục vụ coredns replica 1.

- Shim 4: Phục vụ coredns replica 2.

Mỗi -id (ví dụ: cb8e9659...) tương ứng với một container ID cụ thể. Bạn có thể dùng lệnh `crictl inspect <ID>` để xem chi tiết từng cái.
