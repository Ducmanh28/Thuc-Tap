# Pod trong K8S
Trong K8s, ta không triển khai trực tiếp Container mà triển khai Pod. Pod đóng vai trò như một "Logical Host" (Máy chủ logic), bao bọc một hoặc nhiều Container có liên kết chặt chẽ với nhau, chạy trên cùng một Node và chia sẻ chung một ngữ cảnh (namespaces, cgroups).
## Đặc tính Thiết kế (Design Principles)
Tính phù du: Pod sinh ra để "chết". Khi Node sập hoặc thiếu tài nguyên (Eviction), Pod sẽ bị hủy thay vì di chuyển.

Tính bất biến: Phần lớn Metadata và Spec của một Pod đang chạy không thể sửa đổi (trừ một số ngoại lệ như thêm Ephemeral Containers để debug hoặc Resize tài nguyên). Để update app, hệ thống sẽ tạo Pod mới dựa trên Template mới thay vì patch Pod cũ.

Mô hình triển khai:
- Single-container: 1 Pod bọc 1 Container
- Multi-container: Gom nhóm các Container chạy song hành, bổ trợ cho nhau 
## Cơ chế Quản trị
Trong thực tế (Production), kỹ sư DevOps hiếm khi tạo Pod trực tiếp. Pod luôn được định nghĩa thông qua PodTemplate và được quản lý vòng đời bởi các Controllers:
Deployment / StatefulSet: Đảm bảo High Availability (Replication), tự động 
Rollout/Rollback và Self-healing (tự tạo Pod mới thay thế Pod chết).
DaemonSet: Đảm bảo mỗi Node có chính xác một Pod chạy (thường dùng cho Monitor/Logging agent).

Static Pods: Đây là ngoại lệ đặc biệt. Static Pod được quản lý trực tiếp bởi kubelet trên Node mà không cần qua API Server (thường dùng để boot các component của Control Plane như kube-apiserver)
## Cơ chế Chia sẻ Nội bộ
Các Container nằm trong cùng một Pod chia sẻ 2 tài nguyên huyết mạch:
Storage (Volumes): Các Container có thể mount chung một Volume để đọc/ghi file chéo nhau (VD: Container A tạo file tĩnh, Container B serve file đó làm Web).
Networking: Tất cả Container chung một Network Namespace, chia sẻ chung 1 dải IP và Port. Chúng giao tiếp với nhau qua localhost. (Lưu ý: Không được set trùng Port giữa các Container trong cùng 1 Pod).

## Quản trị Tài nguyên & Bảo mật (Resources & SecurityContext)
Resource Requests & Limits: 
Khai báo CPU/RAM để Kube-scheduler có cơ sở xếp lịch (Scheduling).
- Vượt Limit RAM: Kernel sẽ lập tức "bắn bỏ" tiến trình (OOMKilled).
- Vượt Limit CPU: Tiến trình bị bóp băng thông (Throttling), gây chậm app nhưng không chết.

SecurityContext: Phân quyền ở cấp độ Pod/Container. 

Best Practice là chạy app với user non-root (VD: runAsUser: 1000) và giới hạn quyền truy cập file system (fsGroup).
## Mô hình Multi-Container & Cơ chế Khám bệnh (Probes)
Để phân tách trách nhiệm trong một Pod phức tạp, K8s cung cấp các loại Container khác nhau:

|Loại Container |Đặc điểm|
|---|---|
|Init Container|Chạy tuần tự và phải hoàn tất (Exit 0) trước khi App Container khởi động. Dùng để migrate database, check dependencies, đổi quyền file.|
|App Container|Chứa logic chính của ứng dụng.
|Sidecar Container(Từ v1.33)| Bản chất là Init Container nhưng set restartPolicy: Always. Chạy nền song song cùng App chính trong suốt vòng đời Pod (VD: Envoy proxy trong Istio).|
|Ephemeral Container|Bơm "nóng" vào Pod đang chạy để gỡ lỗi (Troubleshooting) khi image gốc không có công cụ shell/debug.|

## Nhóm lệnh kiểm tra chi tiết và Debug
Dùng khi Pod gặp lỗi hoặc bạn cần soi cấu hình runtime.

Xem thông tin chi tiết (Events, IP, Volume, Node):
```
kubectl describe pod <tên-pod>
```
Xem Log của Pod:
```
kubectl logs <tên-pod>
```
Xem Log theo thời gian thực (Follow):
```
kubectl logs -f <tên-pod>
```
Xem Log của một container cụ thể (nếu Pod có nhiều container):
```
kubectl logs <tên-pod> -c <tên-container>
```
Xem Log của Pod trước khi nó bị restart (Previous):
```
kubectl logs <tên-pod> -p
```
## Nhóm lệnh tương tác trực tiếp (Interaction)
Dùng để truy cập vào bên trong Pod hoặc kiểm tra dịch vụ.

Truy cập vào Shell của Pod (Tương tự Docker exec):
```
kubectl exec -it <tên-pod> -- /bin/bash
```
Chạy một lệnh duy nhất bên trong Pod:
```
kubectl exec <tên-pod> -- ls /var/log
```
Port-forward (Ánh xạ cổng từ Pod về máy cá nhân để test nhanh):
```
kubectl port-forward <tên-pod> 8080:80
```
(Sau đó bạn có thể truy cập localhost:8080 trên trình duyệt).

Copy file từ máy Master vào Pod:
```
kubectl cp <đường-dẫn-file-local> <tên-pod>:<đường-dẫn-trong-pod>
```
## Nhóm lệnh quản lý vòng đời (Lifecycle)
Tạo Pod nhanh (Imperative):
```
kubectl run nginx-test --image=nginx
```
Tạo Pod từ file cấu hình (Declarative - Khuyên dùng):
```
kubectl apply -f pod.yaml
```
Xóa Pod:
```
kubectl delete pod <tên-pod>
```
Xóa Pod ngay lập tức (Force delete - không đợi graceful shutdown):
```
kubectl delete pod <tên-pod> --force --grace-period=0
```
Chỉnh sửa trực tiếp cấu hình Pod đang chạy:
```
kubectl edit pod <tên-pod>
```
# Pod Lab Colabnix
## Lab cơ bản
Tạo file Yaml để tạo Pod
```YAML
apiVersion: v1      # apiVersion: Đây là phiên bản của API mà cụm (cluster) sử dụng. 
                    # Khi các phiên bản mới của Kubernetes được phát hành, các chức năng mới sẽ được giới thiệu, 
                    # do đó, các phiên bản API mới có thể được định nghĩa. 
                    # Đối với đối tượng Pod, chúng ta sử dụng phiên bản API là v1.

kind: Pod           # kind: Loại đối tượng mà bạn muốn tạo (ở đây là Pod).
                    
metadata:           # Metadata: Tại đây chúng ta định nghĩa các dữ liệu định danh về đối tượng sắp tạo.
  name: webserver   # Trong ví dụ này, chúng ta chỉ cung cấp tên của Pod. 
                    # Nhưng bạn có thể cung cấp thêm các chi tiết khác như namespace (không gian tên).

spec:               # Spec: Phần này định nghĩa các đặc tính/trạng thái mà đối tượng Kubernetes cần phải có. 
                    # Nhiệm vụ của Cluster là tự động cập nhật trạng thái thực tế của đối tượng 
                    # sao cho luôn khớp với cấu hình mong muốn này. 
                    # Trong ví dụ này, spec hướng dẫn rằng Pod này sẽ có một container với một vài thuộc tính.
  containers:
  - name: webserver  # Tên định danh cho container này.
    image: nginx:latest # Image (ảnh) được dùng để chạy container (lấy từ Registry).
    ports:               # Các cổng (port) sẽ được mở để giao tiếp.
    - containerPort: 80  # Cổng mà container sẽ lắng nghe bên trong Pod.
```

## Label và Selector trong 1 Pod
Để áp dụng nhãn cho Nginx Pod, bạn có thể sử dụng trường metadata.labels trong định nghĩa Pod. Đây là một ví dụ:
```
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
    environment: production
spec:
  containers:
    - name: nginx
      image: nginx:latest
      ports:
        - containerPort: 80
```
Trong ví dụ này, Nginx Pod được gắn nhãn với hai nhãn: `app: nginx` và `environment: production`. Các nhãn này giúp xác định và phân loại Pod dựa trên mục đích và môi trường của nó.

Để chọn Pod dựa trên nhãn của chúng, bạn có thể sử dụng bộ chọn trong các hoạt động Kubernetes khác nhau. Dưới đây là một ví dụ về việc sử dụng bộ chọn trong ReplicaSet để quản lý nhiều Nginx Pods:
```
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:latest
          ports:
            - containerPort: 80
```
Trong ví dụ này, bộ chọn ReplicaSet matchLabels chỉ định rằng nó sẽ chọn Nhóm có ứng dụng nhãn`: nginx`. Điều này đảm bảo rằng ReplicaSet chỉ quản lý Nginx Pods được gắn nhãn `app: nginx`.

Bạn có thể sử dụng bộ chọn nhãn tương tự trong các tài nguyên Kubernetes khác như Dịch vụ, Triển khai và StatefulSets để nhắm mục tiêu các Pod cụ thể dựa trên nhãn của chúng.

Nhãn và bộ chọn là những khái niệm mạnh mẽ trong Kubernetes cho phép nhóm, lọc và nhắm mục tiêu linh hoạt Pod và các tài nguyên khác. Chúng cho phép bạn áp dụng các hoạt động và cấu hình có chọn lọc cho các Pod cụ thể dựa trên nhãn của chúng, giúp quản lý và thay đổi quy mô ứng dụng của bạn dễ dàng hơn.

Để tìm nạp Pod có nhãn cụ thể bằng lệnh kubectl, bạn có thể sử dụng lệnh `kubectl get pods với –selector` hoặc cờ `-l` theo sau là bộ chọn nhãn. Đây là một ví dụ:
```
kubectl get pods -l app=nginx

NAME        READY   STATUS    RESTARTS   AGE
nginx-pod   1/1     Running   0          2m56s
```
Trong ví dụ này, lệnh sẽ tìm nạp tất cả các Pod có ứng dụng nhãn với giá trị nginx. Thay thế app=nginx bằng bộ chọn nhãn mong muốn của bạn để tìm nạp Pod dựa trên nhãn cụ thể của bạn.

Ngoài ra, bạn có thể sử dụng cờ –show-labels để hiển thị nhãn của các Pod đã tìm nạp:
```
kubectl get pods -l app=nginx --show-labels
NAME        READY   STATUS    RESTARTS   AGE     LABELS
nginx-pod   1/1     Running   0          3m57s   app=nginx,environment=production
```
