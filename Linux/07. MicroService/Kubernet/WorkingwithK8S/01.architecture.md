# Kiến Trúc Hệ Thống Kubernetes: Control Plane và Worker Node
Một cụm (cluster) Kubernetes vận hành dựa trên kiến trúc hệ thống phân tán, được thiết kế để quản lý các ứng dụng container hóa (containerized workloads) ở quy mô lớn. Về mặt logic và vật lý, một Kubernetes Cluster được chia thành hai phân hệ cốt lõi: Control Plane (Mặt phẳng điều khiển) và Worker Nodes (Nút thực thi).

Đơn vị tính toán nhỏ nhất được lập lịch và quản lý trên các Node là các Pod (nhóm chứa một hoặc nhiều container chia sẻ chung network và storage).

## Control Plane (Mặt Phẳng Điều Khiển)
Control Plane là "bộ não" của cluster, chịu trách nhiệm đưa ra các quyết định toàn cục (global decisions như scheduling), duy trì trạng thái hệ thống, và phản hồi lại các sự kiện (cluster events). Để đảm bảo tính sẵn sàng cao (High Availability - HA), các thành phần này thường được triển khai trên nhiều máy chủ chuyên trách (Master Nodes).

- `kube-apiserver` (API Gateway): Là điểm giao tiếp trung tâm của toàn bộ K8s cluster. Nó phơi bày **Kubernetes REST API**, tiếp nhận mọi request từ người dùng (qua kubectl) hoặc từ các thành phần nội bộ. API Server thực hiện các tác vụ Authentication (xác thực), Authorization (phân quyền), và Admission Control trước khi ghi nhận thay đổi vào cơ sở dữ liệu. Thành phần này được thiết kế theo kiến trúc stateless, cho phép scale-out theo chiều ngang dễ dàng.

- etcd (Cơ Sở Dữ Liệu Trạng Thái):
Là một key-value store phân tán, nhất quán (strongly consistent) và có tính sẵn sàng cao. etcd là kho lưu trữ duy nhất chứa toàn bộ dữ liệu về trạng thái (cluster state) và cấu hình của K8s. Mọi thay đổi về cấu hình hay trạng thái Pod đều được lưu tại đây và chỉ API Server mới có quyền tương tác trực tiếp với etcd.

- kube-scheduler (Bộ Lập Lịch):
Thành phần này liên tục theo dõi API Server để phát hiện các Pod mới được tạo nhưng chưa được gán (assign) cho bất kỳ Worker Node nào. Nó sử dụng các thuật toán chấm điểm (scoring algorithms) để chọn ra Node tối ưu nhất cho Pod, dựa trên: yêu cầu tài nguyên (CPU/Memory requests & limits), các ràng buộc phần cứng/phần mềm, affinity/anti-affinity specifications (ái lực), và data locality (vị trí dữ liệu).

- kube-controller-manager (Trình Quản Lý Điều Khiển):
Thực thi các tiến trình điều khiển (controller processes) dưới dạng các vòng lặp đối chiếu (reconciliation loops). Các controllers này liên tục so sánh trạng thái hiện tại (current state) của hệ thống với trạng thái mong muốn (desired state) được khai báo, và thực hiện các hành động để đồng bộ chúng. Các controller cốt lõi bao gồm:

  - Node Controller: Giám sát trạng thái health-check của các Node.

  - ReplicaSet/Deployment Controller: Đảm bảo duy trì đúng số lượng Pod replicas.

  - EndpointSlice Controller: Liên kết Services với các Pods tương ứng.

  - cloud-controller-manager (Tùy chọn): Tách biệt logic điều khiển cụ thể của nền tảng đám mây (AWS, GCP, Azure...) khỏi core Kubernetes. Thành phần này chỉ chạy các controllers tương tác trực tiếp với API của Cloud Provider (ví dụ: cấp phát Load Balancer vật lý, định tuyến network cloud, giám sát trạng thái node trên cloud).

## Worker Node (Nút Thực Thi)
Worker Nodes là các máy chủ vật lý hoặc máy ảo (VM) cung cấp môi trường runtime để thực thi các ứng dụng. Mỗi Node chạy các thành phần sau để giao tiếp với Control Plane và vận hành các Pods.

### kubelet (Agent Quản Lý Node):
Là tác nhân (agent) chính chạy trên mọi Node. kubelet nhận các chỉ thị cấu hình Pod (PodSpecs) từ API Server và tương tác với Container Runtime để đảm bảo các containers mô tả trong PodSpec đang thực sự chạy và ở trạng thái khỏe mạnh (healthy). Nó cũng báo cáo ngược lại trạng thái của Node và Pod về cho API Server.

### kube-proxy (Trình Định Tuyến Mạng):
Là một network proxy chạy trên mỗi Node, chịu trách nhiệm hiện thực hóa khái niệm Service trong Kubernetes. Nó duy trì các quy tắc định tuyến mạng (network rules) ở tầng OS (như iptables hoặc IPVS), cho phép forward traffic từ bên trong hoặc bên ngoài cluster đến đúng các Pod backend có IP biến động (ephemeral IPs).

### Container Runtime (Engine Thực Thi):
Phần mềm nền tảng chịu trách nhiệm pull images và chạy các containers. Kubernetes không trực tiếp chạy container mà giao tiếp với các runtimes này thông qua chuẩn giao tiếp Container Runtime Interface (CRI). Các runtimes phổ biến bao gồm containerd và CRI-O 

## Addons (Tiện Ích Cấp Cụm)
Mặc dù hệ thống cơ bản có thể hoạt động với các thành phần trên, một cụm K8s hoàn chỉnh cho production thường yêu cầu thêm các Addons (chạy dưới dạng các Pods đặc quyền):

- Cluster DNS (CoreDNS): Thành phần bắt buộc để Service Discovery hoạt động. Nó tự động cấp phát và phân giải bản ghi DNS cho các Kubernetes Services, giúp các Pods có thể giao tiếp với nhau qua tên miền thay vì IP nội bộ.

- Observability (Giám sát & Log): Tích hợp các hệ thống như Prometheus (thu thập metrics tài nguyên) và Fluentd/Elasticsearch (gom log tập trung ở cấp độ cluster).
